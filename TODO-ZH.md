# AI 模块路线图：Skills + MCP

## 目的

为 CtxRun 的 AI 模块构建一个受 Codex 启发的能力系统，使得：

- 本地工具不再硬编码到单薄的前端循环中
- Skills 成为可复用、可发现、可作用域限定能力包
- MCP 成为受管理的运行时子系统，而非"动态工具"
- 审批、凭证、服务器状态和事件流成为一等公民

本路线图基于：

- 当前 CtxRun 架构，位于 src/lib/agent、src/components/features/spotlight、src/store、src-tauri
- 本地 Codex 参考快照，位于 ctxrun_docs/codex

---

## 核心决策

### 决策 1：Skills 优先，MCP 其次

**原因：**

- Skills 的运行时复杂度更低
- Skills 能立即创造产品价值
- MCP 依赖于配置、审批、密钥存储、状态报告和长连接

**结论：**

- 在 MCP MVP 之前先发布 Skills MVP

---

### 决策 2：不要 1:1 复制 Codex 的协议表面

**原因：**

- CtxRun 目前并非运行完整的应用服务器风格后端
- 克隆所有 JSON-RPC 方法会增加巨大复杂度，却无近期的产品收益

**结论：**

- 复制 Codex 的架构模式，而非其完整的公开协议
- 初期实现保持桌面本地化和产品导向

---

### 决策 3：MCP 运行时必须存在于 Tauri/Rust 中，而非主要在前端 TS 中

**原因：**

- MCP 需要长连接
- OAuth 和令牌存储不应在纯前端状态中处理
- 审批和沙箱决策需要一个可信的执行边界

**结论：**

- 前端负责渲染状态和发起请求
- 后端拥有 MCP 服务器配置、认证、连接生命周期和工具执行

---

### 决策 4：Skills 不是提示词模板

**原因：**

- Codex 将 Skills 视为具有元数据、作用域、策略和依赖的结构化能力单元
- 纯提示词拼接在规模化时会变得难以管理且不安全

**结论：**

- 引入具有元数据和生命周期的真实 Skill 模型
- 将 Skill 指令作为结构化的上下文项注入，而非盲目字符串追加

---

## 当前 CtxRun 的差距

- 当前的 Agent 循环由前端驱动，仍以 messages + 硬编码工具 为中心
- 当前的默认工具策略是固定的允许列表
- 当前的聊天 UI 只有轻量的消息/工具追踪状态
- 当前的 AI 配置将密钥持久化在应用本地 JSON 中
- 当前的工具注册表不建模动态能力、Skill 元数据或 MCP 来源
- 当前的审批支持存在于工具运行时中，但尚未作为统一的 AI 运行时流程

## 目标架构

### 1. 能力层

引入统一的 CapabilityRegistry 抽象，可暴露：

- 内置本地工具
- Skill 提供的能力
- MCP 提供的工具
- 未来的插件/应用能力

每个能力应携带：

- 稳定 ID
- 来源类型：uiltin | skill | mcp | plugin
- 显示元数据
- 输入模式
- 风险/审批元数据
- 启用/禁用状态
- 来源和所属作用域

---

### 2. 轮次运行时层

引入受 Codex items 启发的结构化轮次/事件模型：

- ssistantMessage
- easoning
- 	oolCall
- pprovalRequest
- skillInjection
- mcpServerStatus
- mcpToolCall
- error

目标不是与 Codex 实现完整的协议对等。

目标是停止将所有内容视为原始消息文本，并给 UI 足够的结构以渲染：

- 审批
- 实时工具状态
- Skill 注入
- MCP 服务器连接状态
- 可恢复的多步骤执行

---

### 3. 密钥与配置层

将配置拆分为：

- 非敏感 UI 设置放在现有的持久化应用配置中
- 敏感值放在 OS 密钥环中
- Skills 和 MCP 的结构化运行时配置放在专用配置文件中

**推荐方向：**

- 用户偏好保留在现有的 JSON 支持的存储中
- 添加新的运行时配置文件用于能力设置
- 通过密钥环支持的 Tauri 命令存储提供商 API 密钥和 MCP OAuth 令牌

---

### 4. 审批层

审批决策应支持：

- 允许一次
- 允许本次会话
- 若产品策略允许则持久化规则
- 拒绝

审批应附加到具体的待处理操作项，而不是作为侧通道处理。

---

### 5. MCP 运行时层

后端管理的子系统应拥有：

- 配置加载
- 服务器启动/关闭
- 连接池
- 超时处理
- 工具名称限定
- 认证状态
- 列出工具/资源/资源模板
- 每次调用的审批和遥测

## 成功定义

CtxRun 最终应支持以下顶层体验：

- 用户可以发现可用能力，而不是猜测存在什么
- 用户可以有意图地调用 Skill 并看到它注入了什么
- 用户可以安全地连接 MCP 服务器并检查其状态
- 工具和 MCP 操作具有可见的审批和结果
- 密钥存储在 OS 支持的安全存储中
- AI 模块成为可扩展的运行时，而非硬编码的聊天助手

---

## 补充方向 - 架构优先，前端重度 AI

本节在上述路线图之上添加产品方向指导。

主要战略建议是：

- CtxRun 不应演变为一个带有许多无关页面的通用"AI 工具箱"。
- CtxRun 应演变为桌面原生的开发者操作层。
- 架构应优先于新的 AI 特性。
- 即使可信执行保留在 Tauri/Rust 中，AI 也应感觉前端原生。

实践中，这意味着：

- 用户应通过快速、可检查、多表面的前端运行时体验 AI
- 后端应保持执行、密钥、审批和长期集成的可信控制平面
- 每个现有模块应成为能力图的一部分，而非独立的功能孤岛

---

## 产品论点

CtxRun 已经拥有差异化产品的种子：

- Context Forge 可以收集和打包本地项目状态
- Spotlight 是高频入口点
- Patch Weaver 可以将 AI 输出转换为代码更改
- Refinery 可以提供最近的用户上下文
- Automator 可以在软件和浏览器中执行操作
- Model Miner 可以摄取外部网络知识

这不是六个独立的产品。

它们应汇聚为一个循环：

1. 捕获上下文
2. 用 AI 决策
3. 带审批执行
4. 将结果显示为可复用工件

长期定位应是：

- 不是"另一个聊天 UI"
- 不是"另一个附带 AI 的启动器"
- 不是"另一个编码 Agent 克隆"
- 而是一个可以观察、推理并在工作区、桌面、浏览器和系统工具上行动的本地优先开发者操作层

---

## 为什么 AI 应是前端重度的

出于 UX 原因而非信任原因，产品应在前端重度使用 AI。

**前端应拥有：**

- 对话/会话组合
- 事件渲染和交互
- 能力发现 UI
- Skill 选择和检查
- 上下文预览、修剪和可解释性
- 轮次历史、工件和重放 UX
- 延迟掩盖、流式、乐观渲染和可恢复性
- 跨表面入口点，如 Spotlight、补丁审查和侧抽屉

**后端应拥有：**

- 密钥存储
- 审批执行
- 危险命令执行
- MCP 进程生命周期
- 浏览器和桌面自动化运行时
- OS 集成
- 在信任边界重要的持久本地持久化

规则应是：

- AI 体验在前端
- AI 能力边界在后端

这很重要，因为产品风险不在于 AI"太前端"。真正的风险是 AI 成为不可见的后端机器，体验差且可检查性弱。

---

## 架构定位

**推荐边界：**

- 前端是编排 Shell 和交互运行时
- 后端是能力宿主和可信执行层

避免这两种极端：

- 纯前端 Agent 循环，安全敏感逻辑泄漏到 UI 状态中
- 后端主导的 Agent 运行时，将前端变成薄终端

**按责任推荐划分：**

### 前端职责

- 维护会话状态并渲染结构化轮次时间线
- 在模型调用之前组装用户可见的上下文束
- 为轮次选择并启用 Skills
- 清晰暴露工具来源和审批状态
- 让用户检查注入了什么、调用了什么、改变了什么
- 支持由一个运行时模型支持的多个 AI 表面
- 即使后端调用缓慢，也保持交互速度和连续性

### 后端职责

- 执行已批准的操作
- 解析密钥支持的提供商设置
- 托管 MCP 服务器和连接状态
- 提供可信能力元数据
- 持久化耐用运行时配置
- 发出前端无需猜测即可直接渲染的结构化事件
- 保护系统免受不安全或格式错误的工具执行

---

## 战略产品方向

下一阶段应优先考虑集成而非扩展。

不要主要发布更多独立模块。相反，使当前模块在一个 AI 运行时内可组合。

**高置信度产品赌注：**

### 1. Spotlight 成为主要 AI 入口点

Spotlight 应成为以下内容的通用调用表面：

- ask
- search
- patch
- run
- inspect
- automate
- 重新打开最近的上下文/工件

用户不应首先决定打开哪个页面或功能。

### 2. Skills 成为专业知识的打包层

Skills 应比提示词模板更重要。

它们应打包：

- 指令
- 上下文规则
- 能力依赖
- UI 支持
- 安全期望
- 可复用工作流

Prompt Verse 应逐渐成为 Skill 内容的一个来源，而非永远平行的原语。

### 3. Patch、Context、Miner 和 Refinery 成为上下文通道

这些模块应作为结构化输入提供给 AI 运行时：

- 工作区上下文
- diff 和待处理编辑
- 剪贴板历史
- 抓取的外部知识
- 保存的项目记忆

这比将它们作为具有独立心智模型的隔离工具更有价值。

### 4. Automator 成为操作后端，而不仅仅是独立自动化页面

Automator 具有战略重要性，因为它赋予 CtxRun 真实世界的操作能力。

它最终应服务于三个角色：

- 用户显式编写的工作流
- 带审批的 AI 建议的多步骤操作
- 用于专门任务的 Skill 支持操作原语

### 5. MCP 应被视为生态扩展，而非首要身份

MCP 很重要，因为它标准化了外部能力访问。但 MCP 不应独立成为产品故事。

故事应保持：

- CtxRun 为开发者提供本地优先的 AI 操作层
- MCP 是能力进入该层的方式之一

## 推荐内部模型

系统应逐渐收敛到类似这样的模型：

- Surface
- Session
- Turn
- ContextItem
- Capability
- Action
- Artifact
- Approval
- Event

**建议含义：**

- Surface：Spotlight、完整聊天、补丁视图、automator 助手面板、未来的内联小部件
- Session：在适当时跨表面共享的一个连续用户交互状态
- Turn：一个用户意图及其执行生命周期
- ContextItem：文件、diff、剪贴板项、网络内容、记忆笔记、Skill 注入
- Capability：内置工具、Skill 能力、MCP 工具、插件提供的操作
- Action：带审批、状态和输出的已调用操作
- Artifact：耐用输出，如补丁集、摘要、工作流、笔记、提取的文档
- Approval：附加到具体待处理操作的显式权限决策
- Event：在 UI 中直接渲染的结构化运行时发射

该模型应首先在共享的 TypeScript 类型中表达以提高 UI 速度，然后在 Rust 中镜像，在信任或持久化需要时。

## 前端架构优先级

如果 AI 是前端重度的，前端需要比简单的页面-存储模式更强的内部架构。

**推荐优先级：**

### 优先级 A - 统一 AI 会话存储

创建专用会话域，而不是将 AI 状态分散到功能本地 hooks 和 stores 中。

它应拥有：

- 会话
- 轮次
- 结构化事件
- 待处理审批
- 活跃能力
- 选定 Skills
- 上下文附件
- 轮次期间生成的工件
- 渲染所需的模型/提供商元数据

### 优先级 B - 结构化轮次时间线

不要仅将 AI 交互渲染为普通交替聊天气泡。

UI 应能表示：

- assistant 输出
- 工具开始
- 需要工具审批
- 工具已批准
- 工具完成
- MCP 服务器已连接/断开
- Skill 已注入
- 补丁已生成
- 补丁已应用
- 工作流已建议
- 工作流已执行
- 带可操作恢复的失败

这个时间线是产品优势，而不仅仅是实现细节。

### 优先级 C - 能力资源管理器和调用 UI

用户应能在轮次之前或期间检查可用能力。

**需要的 UI 表面：**

- 能力列表
- 来源/出处
- 风险级别
- 是否需要审批
- 哪个 Skill 或 MCP 服务器引入了它
- 它是否被禁用、缺失或不健康

### 优先级 D - 上下文编辑器

在模型调用之前，用户应能够理解正在发送什么上下文。

编辑器应显示：

- 附加文件
- 推断的 diff
- 剪贴板项
- 挖掘的页面
- 记忆笔记
- 注入的 Skills
- 预估令牌成本
- 由于预算丢弃了什么

### 优先级 E - 工件中心 UX

重要输出应超出短暂答案气泡而存在。

一等工件应包括：

- 补丁提案
- 工作流草稿
- 摘要
- 提取的文档
- 可复用上下文束
- 可复用 Skill 调用预设

---

## 后端架构优先级

即使拥有前端重度的 AI 产品，后端架构仍然首先，因为它定义了安全性和可扩展性。

**推荐优先级：**

### 优先级 1 - 能力宿主

后端应向前端暴露稳定的能力清单。

该宿主应提供：

- 规范能力 ID
- 能力来源类型
- 执行要求
- 审批元数据
- 健康/状态
- 依赖信息
- 机器可读模式

### 优先级 2 - 审批内核

审批逻辑不应跨功能模块复制。

一个审批子系统应管辖：

- exec 运行时
- 补丁应用
- automator 执行
- MCP 工具调用
- 未来插件操作

### 优先级 3 - 密钥和身份边界

所有提供商密钥、OAuth 令牌和其他敏感运行时值应远离普通前端持久化。

前端可以请求使用能力。前端不应拥有密钥的事实来源。

### 优先级 4 - 事件发射器合约

后端应发出前端可以直接渲染的稳定事件流。

这应减少前端猜测，避免临场状态重建。

### 优先级 5 - 长期运行时管理器

长期或状态化子系统应集中在后端管理器中：

- MCP 连接管理器
- 浏览器自动化管理器
- 桌面自动化状态
- 索引/挖掘工作器
- 适用时的后台上下文收集器

## 近期路线图调整

现有路线图阶段仍然有效。然而，执行重点应调整如下：

### 首先

- 完成能力和审批基础
- 定义前端会话和事件模型
- 使 AI 交互可检查，而非使其更自主

### 其次

- 发布具有强 UI 可见性的显式 Skills
- 统一 Spotlight、补丁、miner 和剪贴板流程的上下文组合
- 使工件耐用且可重访

### 第三

- 添加 MCP 作为后端管理的扩展路径
- 在相同的前端能力 UI 中暴露 MCP 工具
- 在调用失败前使依赖状态可见

### 第四

- 让 Automator 参与 Agent 流
- 支持 AI 建议的工作流和步骤预览
- 让用户保持在审查和批准循环中

## 本方向的产品非目标

为保持专注，避免过早过度投入这些：

- 通用网络 SaaS 控制平面
- 多用户企业协作为主要路线图驱动因素
- 无明确可见性的自主后台 Agent
- 在能力质量和信任 UX 强大之前的插件市场
- 与 Codex/OpenHands 风格后端协议表面的完全对等

---

## 本策略的成功标准

如果产品开始感觉像这样，本补充方向就有效了：

- 一个具有多个表面的 AI 运行时，而非许多无关工具
- 一个审批系统，而非每个功能的确认逻辑
- 一个能力模型，而非分散在应用中的硬编码工具列表
- 一个上下文图，而非单独的功能本地附件
- 一个工件流，而非一次性聊天输出

从用户角度来看，产品应感觉：

- 更快进入
- 更易信任
- 更易检查
- 更易复用
- 更难过时
